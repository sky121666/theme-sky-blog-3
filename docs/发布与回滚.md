# 发布与回滚

## 当前发布流程

1. 在本地完成依赖、构建、发布包和最新稳定版 Halo 验证。
2. 创建版本标签并公开 GitHub Release，不手工上传 ZIP；由原有 CD 自动打包并上传对应版本 ZIP。
3. GitHub Release 公开后，原有 `.github/workflows/cd.yaml` 调用 Halo 官方 `theme-cd.yaml@v4`，同步 Halo 应用市场。

Halo 最低版本只由 `theme.yaml` 的 `spec.requires` 声明为 `>=2.25.0`。本项目不为最低版本维护单独运行矩阵，实际适配与发布前验证只面向当前最新稳定版 Halo `2.25.4`。

## 发布前门禁

- `pnpm outdated --format json` 返回 `{}`。
- `pnpm run check`、`pnpm run audit:licenses` 通过。
- 经明确授权后执行 `pnpm run audit:security`，生产与完整依赖图均为 0 条漏洞。
- `pnpm run verify:reload`、Playwright、性能和 PJAX 门禁在最新稳定版 Halo 上通过。
- `node scripts/verify-release-package.mjs <zip>` 通过。
- 仓库 Secret `HALO_PAT` 具备 Halo 应用市场“版本管理”权限。
- `theme.yaml` 与 `.github/workflows/cd.yaml` 的 App ID 均为 `app-gqnoxtpt`。

## 发布后核验

1. GitHub Release 为公开状态，标签、标题、说明和 ZIP 版本一致。
2. 从公开 Release 下载 ZIP，再运行发布包校验。
3. Halo 应用市场出现相同版本，兼容范围为 `>=2.25.0`。
4. 保留上一稳定 Release 与 ZIP 作为回滚基线。

## 回滚原则

- 已发布版本不可覆盖；修复问题时发布更高补丁版本。
- 站点发生严重回归时，先恢复上一稳定 ZIP，再定位问题并发布修复版。
- 不删除或改写已经公开的标签、Release 和应用市场版本。
- GitHub 或 Halo 应用市场发布失败时，先核对精确版本和外部状态，再决定重试；不使用额外自动恢复工作流。

规则依据：

- [Halo 官方发布应用文档](https://docs.halo.run/developer-guide/app-store/publish-app)
- [GitHub 官方 Release 管理文档](https://docs.github.com/en/repositories/releasing-projects-on-github/managing-releases-in-a-repository)
