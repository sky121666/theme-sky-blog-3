# 图库

## 结论

图库是一个独立 App，协议名是：

- `pageApp = photos`
- `pageMode = browser-list`
- `windowVariant = photos`

它不是 explorer 的样式分支，也不是 reader 的特例。

## 插件契约

当前主题对 `PluginPhotos` 拆成两个 surface 记录：

1. `/photos` 列表、分页、分组、精选集：契约版本确认到 `PluginPhotos` 2.1.1，本地 2.1.1 smoke 已通过。
2. `/photos/{name}` 详情：按 `PluginPhotos` 2.1.1 的 `PhotoVo`、`neighbors`、`photoUrl.detail(...)`、`backUrl`、`exif` 和 `spec.tags` 适配。

列表页已消费的关键模板变量：

- `groups`
- `photos`
- `photoUrl.detail(photo)`
- `photos.nextUrl`
- `group`
- `page`
- `size`

## 公共 REST 增强评估

`PluginPhotos` 2.1.1 提供公共 REST 查询能力，但当前主题暂不切换到 REST 渲染。

| 能力 | 本地结果 | 判断 |
| --- | --- | --- |
| `/apis/api.photo.halo.run/v1alpha1/photos` | 200，当前 46 张照片 | 可作为未来无刷新分页或详情预取的数据源 |
| `ungrouped=true` | 200，当前 0 条 | 暂无入口价值 |
| `tag=<name>` | 200，当前没有 tag 样本 | 暂无入口价值 |
| `/apis/api.photo.halo.run/v1alpha1/photogroups` | 200，当前 4 个分组 | SSR/Finder 已覆盖分组展示 |
| `/apis/api.photo.halo.run/v1alpha1/photos/tags` | 404 | tag 列表不是公共端点，不作为主题前台依赖 |

当前结论：

1. 保持 SSR/Finder 和 HTML 下一页加载，不改成全前端 REST 渲染。
2. 详情预取、tag 视图、ungrouped 入口只有在有真实内容和明确体验收益时再做。
3. 未来如果要做 REST 增强，应先补 tag/ungrouped 样本，再用浏览器验证 LCP、CLS、图片 lazy 和 PJAX 状态。

## 视图模式

当前有 3 种主要视图：

1. `library`
   - `/photos`
2. `albums`
   - `/photos?view=albums`
3. `group`
   - `/photos?group=<name>`

## 壳层联动

图库有自己的窗口标题栏协议字段：

- `data-photos-chrome-title`
- `data-photos-chrome-subtitle`

用于：

- 首屏标题
- same-variant 切换后的标题同步
- 浏览器历史回退后的 chrome 恢复

## 关键交互

当前重点交互包括：

- 方格 / 瀑布流
- 每行数量调整
- same-variant 页面切换
- 相册侧边栏
- 亮暗色适配

运行时主入口：

- [/src/apps/photos/runtime/explorer.js](/src/apps/photos/runtime/explorer.js)

## 什么时候改图库

这些需求应该优先改图库 App：

1. 图库卡片样式
2. 标题栏 `+/-`、布局模式按钮
3. 精选集样式
4. 相册侧边栏显示
5. 图库分页与滚动

不要把图库样式重新挂回 explorer。

## 我想配什么，先配哪里

图库本身后台配置不多，最常见的是：

1. 想改相册侧边栏图标：看 [/annotation-setting.yaml](/annotation-setting.yaml) 中 `PhotoGroup`
2. 想改图库内容：优先去 Halo Photos 插件数据层，不在主题设置里配
