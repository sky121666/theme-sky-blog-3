# 验证与审计矩阵

## 结论

这份文档用于执行“克制优化与证据驱动收口”方案。

原则：

1. 先验证和审计，再决定是否修复。
2. 只修真实问题，不为了统一、满分或洁癖重构。
3. 每批变更独立验证、独立提交。

CI/CD 保持项目原有流程，不承担 Halo 最低版本运行矩阵。最低版本是安装限制，实际适配和回归只跟随当前最新稳定版。

这里的 Halo 核心版本与插件 `contractVersion` / `testedVersion` 相互独立。插件已测试版本只按各自 inventory、源码契约、实际插件资源和专项浏览器证据更新，不从 Halo 最低版本声明推导。

资源边界矩阵：

| 脚本 / 资源 | 允许加载场景 | 不应加载场景 | 处理原则 |
| --- | --- | --- | --- |
| `comment-widget.js` | 有 `<halo:comment>` 或评论入口的页面 | 无评论入口页面 | 页面级 gating 或延迟加载 |
| `rag-ui.js` | 真实启用 RAG 的页面 | 普通首页或不相关 App | 不改插件包体，只控制加载范围 |
| `contact-form.iife.js` | 有联系表单页面 | 普通内容页 | 只在误加载时修复 |
| 大图 / 外链图 | 内容真实需要 | 首屏非必要同步加载 | 先确认影响，再局部优化 |

当前本地审计发现：

| 现象 | 判断 | 处理 |
| --- | --- | --- |
| 受监控的 Comment Widget、AI Assistant、Contact Form、Hyperlink Card、Lottery、Restricted Reading、Vote、Shiki 等资源在 10 个采样页按各自注入规则命中 | 主要由插件通过 `<halo:footer />` 全局注入；当前页面性能仍在预算内 | 暂不移除 `<halo:footer />`，持续记录资源数量和真页错误；除非插件提供页面级开关，否则不在主题侧强行拦截 |
| `/photos`、`/moments`、`/equipments` 媒体体积明显高于其他页面 | 来自真实内容图片，且采样页图片 lazy 比例正常 | 不做全局缓存、虚拟列表或强压缩；只在出现首屏性能异常时局部处理 |
| 首页也会加载 Steam 头像等小组件媒体 | 属于当前桌面小组件真实内容展示 | 只在首屏性能异常时处理，不为了“零资源”隐藏真实内容 |
| 编辑器插件样本有 4 张图片指向 `http://192.168.1.23:8090/upload/...` 并连接失败 | 属于文章正文保存的旧局域网内容地址，不是主题或插件脚本请求；插件资源失败、page error 和可操作控制台错误均为 0 | 保留告警并交给内容维护处理，不在主题兼容修复中改写用户正文 |

## 第三批：最小修复边界

| 类型 | 允许修复 | 不做 |
| --- | --- | --- |
| 插件资源误加载 | 页面级 gating、延迟加载 | 改插件内部包体 |
| 性能真实异常 | 只修异常页面 | 全局缓存、虚拟列表、复杂预加载 |
| 样式共享风险 | 明确共享样式来源和引用边界 | 为拆分而拆分 |
| 模板内联混乱 | 迁移重复交互逻辑 | 清空所有动态 CSS 变量或 SSR 初始状态 |
| 视觉偏差 | Dock、Header、图标、壁纸微调 | 堆强玻璃、重阴影、过度动画 |

## 第四批：macOS 质感微调

| 区域 | 调整目标 | 禁止项 |
| --- | --- | --- |
| Dock | 图标填充、间距、比例、轻量玻璃 | 重阴影、强玻璃、过度放大 |
| Header | 顶级菜单文字化，二级菜单保留官方 icon | 顶级菜单乱加图标 |
| 图标 | 高质量 PNG/位图，填满 Dock 容器 | 低质测试图、无意义 SVG |
| 壁纸 | 真实或高质量预设 | 大段 base64 内置背景 |
| 暗亮切换 | 保持当前平滑切换 | 强制所有独立 App 参与变色 |
