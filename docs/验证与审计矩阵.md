# 验证与审计矩阵

## 结论

这份文档用于执行“克制优化与证据驱动收口”方案。

原则：

1. 先验证和审计，再决定是否修复。
2. 只修真实问题，不为了统一、满分或洁癖重构。
3. 每批变更独立验证、独立提交。

## 第一批：真实样本复验

| 对象 | 要验证 | 当前状态 | 成功标准 | 后续动作 |
| --- | --- | --- | --- | --- |
| Docsme | 权限态、多语言、多版本、Shiki、KaTeX、Mermaid | 已补专项脚本，富内容样本待补 | 页面可访问，PJAX、目录、评论、暗亮色正常 | 运行 `pnpm run verify:docsme`，必要时用 `DOCSME_*_SAMPLE_PATH` 指定样本 |
| Moments | 真实代码块、已审核评论、媒体混排 | 评论样本已通过，代码块样本待补 | 列表和详情渲染正常，评论和媒体不破版 | 运行 `pnpm run verify:moments`，必要时用 `MOMENTS_*_SAMPLE_PATH` 指定样本 |
| Shiki | 文章、Docsme、Moments 的代码高亮 | 文章页已复验，Docsme/Moments 待样本 | 暗亮色切换、PJAX 往返后不重复、不丢失 | 补样本后做跨页面复验 |
| Photos | 公共 REST、tag、ungrouped、详情预取是否有实际收益 | 后置增强评估 | 只确认是否值得做，不默认实现 | 先看 API 收益，再决定是否进入实现 |
| Equipment | 是否进入组件中心 | 产品决策待定 | 有明确展示价值再做组件 | 先判断要不要，不先写组件 |

## 第二批：性能与插件资源审计

执行命令：

```bash
AUDIT_BASE_URL=http://localhost:8090 pnpm run audit:real-pages
```

输出文件：

| 文件 | 用途 |
| --- | --- |
| `output/audit/real-pages-report.json` | 机器可读审计数据 |
| `output/audit/real-pages-report.md` | 人工复核表格 |

专项复验命令：

| 命令 | 用途 |
| --- | --- |
| `pnpm run verify:moments` | 复验 Moments 公开 API、代码块样本、已审核评论样本 |
| `pnpm run verify:docsme` | 复验 Docsme 项目大厅、正文页、代码块、KaTeX、Mermaid 样本 |

样本路径变量：

| 变量 | 用途 |
| --- | --- |
| `MOMENTS_CODE_SAMPLE_PATH` | 指定带代码块的 Moments 详情页 |
| `MOMENTS_COMMENT_SAMPLE_PATH` | 指定带已审核评论的 Moments 详情页 |
| `DOCSME_DOC_SAMPLE_PATH` | 指定 Docsme 正文页 |
| `DOCSME_CODE_SAMPLE_PATH` | 指定带代码块的 Docsme 正文页 |
| `DOCSME_KATEX_SAMPLE_PATH` | 指定带 KaTeX 的 Docsme 正文页 |
| `DOCSME_MERMAID_SAMPLE_PATH` | 指定带 Mermaid 的 Docsme 正文页 |

页面采样矩阵：

| 页面 | 采样内容 | 判断方式 |
| --- | --- | --- |
| `/links` | LCP / CLS / TBT / 插件脚本 | 看评论、友链申请相关资源是否只在需要时加载 |
| `/douban` | LCP / CLS / TBT / 图片与 API 请求 | 看封面、公开 API 是否造成首屏波动 |
| `/steam` | LCP / CLS / TBT / 热力图与 API 请求 | 看热力图、游戏库请求是否影响首屏 |
| `/docs`、`/docs/**` | LCP / CLS / TBT / Shiki/评论资源 | 看代码高亮和评论资源是否符合页面内容 |
| `/equipments` | LCP / CLS / TBT / 图片加载 | 看图片加载是否符合首屏需要 |
| `/moments` | 媒体、评论、通知相关资源 | 看媒体和评论资源是否按内容加载 |
| `/photos` | 图片查看器、图库布局、懒加载 | 看图片数量和 lazy 比例 |

资源边界矩阵：

| 脚本 / 资源 | 允许加载场景 | 不应加载场景 | 处理原则 |
| --- | --- | --- | --- |
| `comment-widget.js` | 有 `<halo:comment>` 或评论入口的页面 | 无评论入口页面 | 页面级 gating 或延迟加载 |
| `rag-ui.js` | 真实启用 RAG 的页面 | 普通首页或不相关 App | 不改插件包体，只控制加载范围 |
| `contact-form.iife.js` | 有联系表单页面 | 普通内容页 | 只在误加载时修复 |
| 大图 / 外链图 | 内容真实需要 | 首屏非必要同步加载 | 先确认影响，再局部优化 |

当前本地审计发现：

| 现象 | 判断 | 处理 |
| --- | --- | --- |
| `comment-widget.js`、`rag-ui.js`、`contact-form.iife.js`、`shiki-code.js` 在 8 个采样页均命中 | 更像插件通过 `<halo:footer />` 全局注入；当前页面性能仍可接受 | 暂不移除 `<halo:footer />`，先记录证据；除非确认插件支持页面级开关，否则不在主题侧强行拦截 |
| `/photos`、`/moments`、`/equipments` 媒体体积明显高于其他页面 | 来自真实内容图片，且采样页图片 lazy 比例正常 | 不做全局缓存、虚拟列表或强压缩；只在出现首屏性能异常时局部处理 |
| 首页也会加载 Steam 头像等小组件媒体 | 属于当前桌面小组件真实内容展示 | 只在首屏性能异常时处理，不为了“零资源”隐藏真实内容 |

## 第三批：最小修复边界

| 类型 | 允许修复 | 不做 |
| --- | --- | --- |
| 插件资源误加载 | 页面级 gating、延迟加载 | 改插件内部包体 |
| 性能真实异常 | 只修异常页面 | 全局缓存、虚拟列表、复杂预加载 |
| 样式共享风险 | 明确共享样式来源和引用边界 | 为拆分而拆分 |
| 模板内联混乱 | 迁移重复交互逻辑 | 清空所有动态 CSS 变量或 SSR 初始状态 |
| 视觉偏差 | Dock、Header、图标、壁纸微调 | 堆强玻璃、重阴影、过度动画 |

## 第四批：macOS 质感微调

| 区域 | 调整目标 | 禁止项 |
| --- | --- | --- |
| Dock | 图标填充、间距、比例、轻量玻璃 | 重阴影、强玻璃、过度放大 |
| Header | 顶级菜单文字化，二级菜单保留官方 icon | 顶级菜单乱加图标 |
| 图标 | 高质量 PNG/位图，填满 Dock 容器 | 低质测试图、无意义 SVG |
| 壁纸 | 真实或高质量预设 | 大段 base64 内置背景 |
| 暗亮切换 | 保持当前平滑切换 | 强制所有独立 App 参与变色 |
