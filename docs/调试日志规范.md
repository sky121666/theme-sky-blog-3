# 调试日志规范

## 结论

调试日志只服务排障，不做行为埋点，也不追求覆盖所有代码路径。默认由后台 `developer.debug_mode` 控制，关闭后前端不输出调试日志。

核心目标是：**报错时给开发者即时提醒，并提供足够的定位线索，让人或 AI 能快速判断下一步检查什么。**

## 开关

| 来源 | 作用 |
| --- | --- |
| `settings.yaml` 的 `developer.debug_mode` | 后端配置入口 |
| `body[data-debug="true"]` | 模板下发到前端的运行时开关 |
| `createLogger()` / `warnApiCall()` | 前端统一日志工具 |

## 应该写日志的位置

| 类型 | 规则 | 示例 |
| --- | --- | --- |
| API 失败 | 必须记录接口、状态或错误摘要 | 插件 API 失败、分页失败、权限接口失败 |
| 状态降级 | 必须记录降级原因和兜底动作 | 自动补全降级为手动草稿、提交失败切换留言兜底 |
| 关键组件异常 | 必须记录组件、失败位置和下一步检查建议 | 豆瓣小组件数据失败、通知已读失败 |
| 结构异常 | 只在会影响渲染或需要重新挂载时记录 | PJAX 后 DOM 未完整水合 |

## 不应该写日志的位置

| 类型 | 原因 |
| --- | --- |
| 每个底层 `fetch` 的 start/success | 噪声大，排障价值低 |
| 成功加载、分页完成、到达末尾 | 属于正常状态，不是错误提醒 |
| 匿名访问、非管理员访问 | 属于预期权限状态，不应刷日志 |
| 缓存 miss / hit | 除非正在排缓存问题，否则会刷屏 |
| 普通点击、hover、滚动 | 不属于异常或状态收口 |
| 大对象完整输出 | 容易泄露内容，也难读 |
| token、cookie、完整用户资料 | 敏感信息禁止输出 |

## 日志内容格式

| 字段 | 建议 |
| --- | --- |
| 前缀 | 使用功能名，如 `[links]`、`[douban]`、`[steam]` |
| 文案 | 面向排障，优先中文，例如“友链申请提交失败，切换留言兜底” |
| payload | 只放 endpoint、status、page、message、action、hint 等摘要字段 |
| action | 当前代码已经采取的兜底动作，例如 `enable-message-fallback` |
| hint | 给开发者和 AI 的下一步排查方向 |
| error | 使用 `error.message`，不要直接展开完整异常对象，除非需要堆栈 |

## 当前重点覆盖

| 模块 | 重点日志 |
| --- | --- |
| Links | 自动补全权限检测异常、站点信息提取失败、提交失败后留言兜底 |
| Douban | 公开 API 异常、列表失败、下一页失败 |
| Photos / Steam / Bangumis / Equipments | 分页失败、接口失败 |
| Moments | 通知加载失败、已读标记失败 |
| Douban Widget | 数据加载失败、缓存异常、DOM 未完整水合 |
